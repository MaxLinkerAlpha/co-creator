## 🎯 AI 快速参考

生成代码前，问自己以下问题：

| 检查项 | 要求 |
| --- | --- |
| **编码** | 所有文件必须使用 `UTF-8` |
| **样板代码** | 拒绝重复生成！复杂修复逻辑必须抽象为独立模块并 `import` |
| **路径处理** | 使用现代路径库（如 Python 的 `pathlib`）而非字符串拼接 |
| **异常处理** | 禁止裸 `except:`，必须捕获具体异常 |
| **并发安全** | 多线程/多进程写入必须使用锁 |
| **文件修改** | 根据复杂度选择 `Fast Mode` (直接写) 或 `Strict Mode` (备份+AST验证) |
| **密钥管理** | 禁止硬编码，必须使用环境变量 (`.env`) |

## 🌐 零号法则：全局通用基准

> **适用场景**: 所有项目｜所有编程语言 (Python/JS/Go 等)｜跨平台

* **绝对的 UTF-8 编码** 所有的源码文件、文本读写（`open(file, encoding='utf-8')`）、网络请求载荷、序列化（JSON/XML），必须强制且唯一地使用 `UTF-8` 编码。严禁使用系统默认编码或区域性编码（如 GBK、Windows-1252），除非用户明确指明是为了兼容遗留系统。
* **语言无关的跨平台兼容** 路径拼接必须使用目标语言现代的路径库（如 Python 的 `pathlib`，Node.js 的 `path.join`），严禁使用硬编码的字符串拼接（如 `dir + "\\" + file`），确保代码在 Windows/Linux/macOS 上表现一致。
* **规范** 禁止在文件名中使用小数点 .（后缀名除外），必须使用下划线 _ 分隔版本号。防止 Python 的 import 语句解析歧义。示例：✅ base_utils_v1_0.py | ❌ base_utils_v1.0.py

## 🏗️ 架构演进：项目复杂度分层原则

> **适用场景**: 工作流决策｜代码生成规模

AI 必须根据用户的当前开发阶段，选择合适的工程复杂度，**严禁对极简脚本过度设计**：

* **Fast Mode (MVP/单脚本探索期)**：
* **特征**：用户要求写“一个简单的抓取测试”或“单文件处理脚本”。
* **行为**：跳过复杂的模块拆分，不使用 AST 语法树检查和备份流。重点在于**逻辑跑通**和**尽早失败 (Fail Fast)**。


* **Strict Mode (生产级/中大型项目)**：
* **特征**：项目已具有特定目录结构，或用户明确要求“健壮性”、“生产可用”。
* **行为**：严格执行依赖注入、关注点分离、多模块拆分，且文件 IO 必须走完整的“备份 -> 验证 -> 修改 -> 回滚”安全流。

## 🏆 黄金法则：样板代码组件化 (以 Windows 控制台修复为例)

> **适用场景**: CLI 工具｜初始化配置

**【核心禁令】** 绝对禁止在每次生成新脚本时，重复输出数十行的“魔法修复代码”或“配置初始化代码”。这既浪费 Token，又违背单一数据源原则。

**【最佳实践：Python Windows 编码修复】**
Windows 命令行默认使用 GBK 编码，导致输出中文或 Emoji 时出现 `UnicodeEncodeError`。AI 生成的 CLI 工具必须自包含编码修复，且**禁止要求用户修改系统注册表**。

在项目初始化阶段，AI 必须主动生成一个 `console_utils.py`（或类似的基础库文件），将修复逻辑**一次性封装**：

```python
# console_utils.py
import sys
import os

def setup_windows_console():
    """Windows 控制台 UTF-8 强制修复 (仅在项目入口调用一次)"""
    if sys.platform == 'win32':
        os.environ['PYTHONIOENCODING'] = 'utf-8' # 影响子进程
        import io
        sys.stdout = io.TextIOWrapper(sys.stdout.buffer, encoding='utf-8', errors='replace') #
        sys.stderr = io.TextIOWrapper(sys.stderr.buffer, encoding='utf-8', errors='replace') #
        import ctypes
        kernel32 = ctypes.windll.kernel32
        kernel32.SetConsoleMode(kernel32.GetStdHandle(-11), 7) # 启用虚拟终端

```

随后，在生成的所有 CLI 脚本中，只需极简引入：

```python
# main.py
from console_utils import setup_windows_console
setup_windows_console()

```

## 🔴 核心架构原则

> **适用场景**: 架构设计

在任何代码生成任务中，AI 必须将以下原则置于最高优先级：

* **KISS（Keep It Simple, Stupid）优先** 永远提供最简单、最直接的解决方案。除非明确要求，否则禁止过度抽象（如毫无必要的工厂模式、多重继承、过度嵌套的类）。
* **YAGNI（You Aren't Gonna Need It）禁令** 只针对当前明确的需求编写代码。禁止为"未来可能的需求"提前预留复杂的扩展接口或冗余参数。
* **单一数据源（Single Source of Truth, SSOT）** 确保同一个状态或数据在系统中只保存在一个地方。禁止在多个类或模块中维护可能产生冲突的重复状态。配置项必须集中管理。
* **单一职责（Single Responsibility）强制** * 一个函数只做一件事，长度尽量控制在 50 行以内
* 遇到超过 100 行的"上帝类（God Class）"或"上帝函数"，AI 必须主动提出拆分建议

## 🛡️ 可靠性与安全性

> **适用场景**: 并发｜安全

* **并发安全（Concurrency Safety）** * 在多线程/多进程/协程环境下，但凡涉及到对同一文件或内存变量的并发写入，AI 必须主动引入并正确使用锁机制（如 `threading.Lock()`、`asyncio.Lock()`）
* 严禁由于未加锁导致的竞态条件（Race Condition）和数据损坏


* **防御性编程（Defensive Programming）与尽早失败** * 在函数入口处优先校验外部输入（参数类型、边界值），不合法则立刻抛出异常（Fail Fast）
* 严禁吞噬异常：绝对禁止使用无上下文的 `except:` 或 `catch (e) { pass }`。必须捕获具体的异常类型，并进行重试、日志记录或合理降级


* **凭证与密钥安全（Secrets Management）** 严禁在代码中硬编码任何 API Keys、数据库密码或 Token。必须通过环境变量（Environment Variables）或外部配置文件（如 `.env`）读取。

## 🟡 文件操作安全规范 (分级策略)

> **适用场景**: 文件 IO｜编码安全

### 1. 安全读取基线 (所有模式适用)

```python
# 必须使用的读取模式，防止编码损坏
with open(file, 'r', encoding='utf-8', errors='replace') as f:
    content = f.read().replace('\ufffd', '')  # 清理替换字符

```

### 2. 高危操作禁令

| 禁止项 | 原因 |
| --- | --- |
| ❌ 在二进制模式（`rb`/`wb`）下直接进行字符串替换 | 会破坏多字节字符 |
| ❌ 使用 `errors='ignore'` | 会导致数据静默丢失 |

### 3. Strict Mode 修改工作流 (针对生产级/Python代码库修改)

如果在对已有的重要源码文件进行重构：

1. **备份** —— 修改前自动创建 `.bak` 备份
2. **语法验证** —— 使用 `ast.parse()` (Python) 或相似工具验证语法完整性
3. **安全修改** —— 确认文件为 UTF-8 后修改写入。
4. **事后验证** —— 若修改后语法不通，立即触发回滚。

## 🟢 代码整洁与可维护性

> **适用场景**: 代码风格｜注释

* **类型注解（Type Hints）强制开启** 在支持的语言（如 Python、TypeScript、Go）中，必须为函数签名、类属性和复杂变量加上完整的类型提示，以增强静态检查和可读性。
* **纯函数（Pure Functions）优先** 数据清洗、转换、计算等逻辑层，应尽量设计为无副作用的纯函数（相同输入永远产生相同输出，不修改全局变量）。
* **依赖注入（Dependency Injection）** 避免在类内部硬编码实例化外部核心组件（如数据库连接、配置对象、日志器）。应通过构造函数将依赖项传入，以保证代码的解耦和可测试性。
* **命名与注释** * 严格遵守目标语言的标准命名约定（如 Python 的 `snake_case`，JS/TS 的 `camelCase`）
* **注释的哲学**：不要解释"在做什么（What）" —— 清晰的变量和函数名就是最好的说明书。必须解释"为什么这么做（Why）" —— 说明业务妥协、特殊边缘情况处理或反直觉逻辑背景。

## 🔵 项目结构与协作

> **适用场景**: 项目初始化｜团队协作

* **关注点分离（Separation of Concerns）** 严格隔离逻辑层。网络请求（API）、数据持久化（DB/File）、核心业务逻辑（Domain）和用户界面（UI/CLI）必须拆分到不同的模块中。
* **版本与文件命名规范** 禁止在代码库中使用 `_final.py` 或 `_new.py` 等非工程化后缀。重大迭代请建立新模块版本（如 `v2` 文件夹）或依赖 Git 记录。

## 📚 经验教训库

> **适用场景**: 风险预防｜历史经验

### 2026-03: 文件编码与批量修改安全

* **风险**: UTF-8 多字节字符（中文、Emoji）在批量替换时极易被破坏，导致文件不可解析（出现 `U+FFFD` 替换字符）
* **对策**: 批量修改必须使用文本模式（`encoding='utf-8', errors='replace'`）；特别警惕字符：✅ ❌ ⚠️ 🎮 📊 等 Emoji 及框线符号。

### 2026-02: 跨平台脚本兼容性

* **风险**: Windows PowerShell 不支持 Unix 的 `&&` 链式操作和 heredoc 语法
* **对策**: 跨平台命令使用分号 `;` 分隔而非 `&&`；复杂逻辑必须写入 `.py` 脚本文件，禁止在 shell 中嵌入多行 Python 代码。
